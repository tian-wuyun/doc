1 引言

1.1 什么是 GitHub Flavored Markdown？

GitHub Flavored Markdown，通常缩写为 GFM，是 Markdown 的方言 当前支持 GitHub.com 和 GitHub 上的用户内容 企业。

这个正式规范基于 CommonMark 规范，定义了语法和 这种方言的语义。

GFM 是 CommonMark 的严格超集。中支持的所有功能 GitHub 用户内容，以及原始 CommonMark 规范中未指定的内容 因此，它们被称为 扩展 ，并因此突出显示。

虽然 GFM 支持广泛的输入，但值得注意的是，GitHub.com 和 GitHub Enterprise 在 GFM被转换为HTML，以确保网站的安全性和一致性。

1.2 什么是降价？

Markdown 是一种用于编写结构化文档的纯文本格式， 基于指示电子邮件格式的约定 和Usenet帖子。 它是由约翰·格鲁伯（John Gruber）开发的（与 来自 Aaron Swartz 的帮助），并于 2004 年以 语法说明 和 Perl 脚本 （ Markdown.pl） 将 Markdown 转换为 HTML中。 在接下来的十年中，数十个实施是 以多种语言开发。 一些人扩展了原来的 具有脚注、表格和 其他文档元素。 有些允许 Markdown 文档 以 HTML 以外的格式呈现。 像Reddit这样的网站， StackOverflow 和 GitHub 有数百万人使用 Markdown。 Markdown 开始在网络之外被用于创作书籍， 文章、幻灯片、信件和讲义。

Markdown 与许多其他轻量级标记的区别 语法通常更容易编写，是它的可读性。 正如格鲁伯所写：

Markdown 格式语法的首要设计目标是 使其尽可能具有可读性。这个想法是 Markdown 格式的文档应按原样发布，如 纯文本，看起来不像是用标签标记的 或格式说明。 （ http://daringfireball.net/projects/markdown/ ）

这一点可以通过比较 AsciiDoc 和 Markdown 的等效样本。 下面是一个示例 AsciiDoc 手册中的 AsciiDoc： 

1. List item one.
+
List item one continued with a second paragraph followed by an
Indented block.
+
.................
$ ls *.sh
$ mv *.sh ~/tmp
.................
+
List item continued with a third paragraph.

2. List item two continued with an open block.
+
--
This paragraph is part of the preceding list item.

a. This list is nested and does not require explicit item
continuation.
+
This paragraph is part of the preceding list item.

b. List item b.

This paragraph belongs to item two of the outer list.
--

这是 Markdown 中的等价物： 

1.  List item one.

    List item one continued with a second paragraph followed by an
    Indented block.

        $ ls *.sh
        $ mv *.sh ~/tmp

    List item continued with a third paragraph.

2.  List item two continued with an open block.

    This paragraph is part of the preceding list item.

    1. This list is nested and does not require explicit item continuation.

       This paragraph is part of the preceding list item.

    2. List item b.

    This paragraph belongs to item two of the outer list.

可以说，AsciiDoc 版本更容易编写。 你不需要 担心缩进。 但是 Markdown 版本要容易得多 阅读。 列表项的嵌套在 源，而不仅仅是在处理后的文档中。

2 预赛

2.1 字符和线条

任何 字符 序列都是有效的 CommonMark 公文。

字符 是 Unicode 码位。 虽然有些 代码点（例如，组合重音符号）不对应于 直观意义上的字符，所有代码点都算作字符 就本规范而言。

此规范不指定编码; 它认为行是组合的 字符 而不是字节。 符合要求的解析器可能会受到限制 到一定的编码。

一 行 是零个或多个 字符 的序列 除换行符 （ U+000A） 或回车 （ U+000D）， 后跟 一行结尾 或文件末尾。

以换行符 （ U+000A），回车 （ U+000D） 后跟换行符或回车符和 跟随换行符。

不包含字符的行，或仅包含空格的行 （ U+0020） 或制表符 （ U+0009），称为 空行 。

此规范中将使用以下字符类定义：

空格字符 是空格 （ U+0020）、制表符 （ U+0009）、换行符 （ U+000A）、线制表 （ U+000B）， 表单馈送 （ U+000C）或回车 （ U+000D）。

空格 是一个或多个 空格的序列 字符 。

Unicode 空格字符 是 Unicode 中的任何代码点 Zs 常规类别或选项卡 （ U+0009）， 回车 （ U+000D）、换行符 （ U+000A），或表单提要 （ U+000C）。

Unicode 空格 是一个序列 或更多 Unicode 空格字符 。

空 格 是 U+0020。

非空格字符 是任何字符 那不是 空格字符 。

ASCII 标点符号字符 是 !， "， #， $， %， &， '， (， )， *， +， ,， -， .， / （U+0021–2楼）， :， ;， <， =， >， ?， @ （U+003A–0040）， [， \， ]， ^， _， ` （U+005B–0060）， {， |， }或 ~（U+007b–007a）。

标点符号 是 ASCII 标点符号字符 或任何内容 常规 Unicode 类别 Pc， Pd， Pe， Pf， Pi， Po或 Ps。

3 块和内联

我们可以将文档视为 块 - 结构元素，如段落、块 引号、列表、标题、规则和代码块。 一些块（如 块引号和列表项）包含其他块;其他（如 标题和段落）包含 内联 内容 - 文本、 链接、强调的文本、图像、代码跨度等。

3.1 先例

块结构的指标始终优先于指标 的内联结构。 因此，例如，下面是一个列表，其中包含 两个项目，而不是一个包含代码跨度的项目的列表：

实施例 12 
- `one
- two`

<ul>
<li>`one</li>
<li>two`</li>
</ul>

这意味着解析可以分两步进行：首先，块 可以辨别文件的结构; 二、里面的文字行 段落、标题和其他块结构可以解析为内联 结构。 第二步需要有关链接引用的信息 仅在第一个末尾可用的定义 步。 请注意，第一步需要按顺序处理行， 但第二个可以并行化，因为内联解析 一个块元素不会影响任何其他块元素的内联解析。

4.2 ATX 标题

ATX 航向 由一串字符组成，解析为内联内容，位于 1-6 个未转义的开场顺序 # 字符和可选 任意数量的未转义的关闭序列 # 字符。 开场顺序 # 字符后必须跟一个 空格 或行尾。可选的关闭顺序 #s 必须是 前面有一个 空格 ，后面只能有一个空格。 开幕式 # 字符可以缩进 0-3 个空格。 的原始内容 标题在解析之前会去除前导和尾随空格 作为内联内容。 标题级别等于 # 开场序列中的字符。

简单标题：

实施例 32 
# foo
## foo
### foo
#### foo
##### foo
###### foo

<h1>foo</h1>
<h2>foo</h2>
<h3>foo</h3>
<h4>foo</h4>
<h5>foo</h5>
<h6>foo</h6>

六个以上 # characters 不是标题：

实施例 33 
####### foo

<p>####### foo</p>

之间至少需要一个空格 # 字符和 标题的内容，除非标题为空。 请注意，许多 实现目前不需要空间。 但是， 空间是 原始 ATX 实现 ， 它有助于防止将以下内容解析为 标题：

实施例 34 
#5 bolt

#hashtag

<p>#5 bolt</p>
<p>#hashtag</p>

这不是一个标题，因为第一个 # 已转义：

实施例 35 
\## foo

<p>## foo</p>

内容被解析为内联：

实施例 36 
# foo *bar* \*baz\*

<h1>foo <em>bar</em> *baz*</h1>

前导和尾随 空格 在解析内联内容时，会忽略

实施例 37 
#                  foo                     

<h1>foo</h1>

允许使用一到三个空格缩进：

实施例 38 
 ### foo
  ## foo
   # foo

<h3>foo</h3>
<h2>foo</h2>
<h1>foo</h1>

四个空格太多了：

实施例 39 
    # foo

<pre><code># foo
</code></pre>
实施例 40 
foo
    # bar

<p>foo
# bar</p>

结束序列 # 字符是可选的：

实施例 41 
## foo ##
  ###   bar    ###

<h2>foo</h2>
<h3>bar</h3>

它的长度不必与开头序列相同：

实施例 42 
# foo ##################################
##### foo ##

<h1>foo</h1>
<h5>foo</h5>

结束序列后允许有空格：

实施例 43 
### foo ###     

<h3>foo</h3>

序列 #之外什么都没有的字符 后面除了空格 不是结束序列，但算作 标题：

实施例 44

HTML 块 是一组经过处理的行 作为原始 HTML（并且不会在 HTML 输出中转义）。

首先，一些定义。 分隔符运行 是 一个或多个序列 * 前面没有的字符或 后跟一个非反斜杠转义 * 字符或序列 一个或多个 _ 前面或后面没有字符 非反斜杠转义 _ 字符。

  1. 应尽量减少嵌套的数量。因此，例如， 解释 <strong>...</strong> 总是首选 <em><em>...</em></em>。

  2. 解释 <em><strong>...</strong></em> 总是 首选 <strong><em>...</em></strong>。

6.8 自动链接

自动链接 是内部的绝对 URI 和电子邮件地址 < 和 >。它们被解析为链接，带有 URL 或电子邮件地址 作为链接标签。

URI 自动链接 由 <，后跟 绝对 URI 后跟 >。 它被解析为 指向 URI 的链接，其中 URI 作为链接的标签。

绝对 URI， 出于这些目的，由 scheme 后跟冒号 （ :） 后跟除 ASCII 以外的零个或多个字符 空格 和控制字符， <和 >。 如果 URI 包含这些字符，它们必须采用百分比编码 （例如 %20 对于一个空格）。

就本规范而言， 方案 是任何序列 2-32 个字符，以 ASCII 字母开头，后跟 通过 ASCII 字母、数字或符号加号的任意组合 （“+”）、句点 （“.”） 或连字符 （“-”）。

概述

在处理的每个点上，文档都表示为 块 。 树的根是 document 块。 这 document 作为孩子可能有任意数量的其他块 children。 这些孩子 反过来，可能在儿童时期有其他块。 块的最后一个子项 通常被认为是 开放 的，这意味着后续的输入行 可以更改其内容。 （未打开的块是 已关闭 。 例如，这里是一个可能的文档树，带有打开的块 用箭头标记：

第 1 阶段：块结构

处理的每一行都对此树产生影响。 这条线是 分析，并根据其内容，文档可能会被更改 通过以下一种或多种方式：

  1. 一个或多个开放区块可以关闭。
  2. 可以创建一个或多个新块作为 最后一个打开的块。
  3. 文本可以添加到剩余的最后一个（最深）打开的块中 在树上。

一旦以这种方式将一条线合并到树中， 它可以被丢弃，因此可以在流中读取输入。

对于每一行，我们遵循以下过程：

  1. 首先，我们遍历开放块，从 根文档，并从最后一个子项下降到最后一个子项 打开块。 每个块都施加了行必须满足的条件 如果块要保持打开状态。 例如，块报价需要 > 字符。 段落需要非空行。 在这个阶段，我们可能会匹配全部或部分开放 块。 但是我们还不能关闭不匹配的区块，因为我们可能有一个 懒惰的延续线 。

  2. 接下来，在消耗现有延续标记之后 块，我们寻找新的块开始（例如 > 用于块报价）。 如果我们遇到新的区块启动，我们会关闭任何不匹配的区块 在将新块创建为上一个块的子块之前的步骤 1 中 匹配块。

-> document

我们经文的第一行， 

> Lorem ipsum dolor

导致 block_quote 块将创建为我们的子级 打开 document 块和 paragraph 块作为子 这 block_quote。 然后，文本将添加到最后一个打开的文本中 块， paragraph： 

-> document
  -> block_quote
    -> paragraph
         "Lorem ipsum dolor"

下一行， 

sit amet.

是开放的“懒惰延续” paragraph，所以它被添加 到段落的正文： 

-> document
  -> block_quote
    -> paragraph
         "Lorem ipsum dolor\nsit amet."

第三行， 

> - Qui *quodsi iracundia*

导致 paragraph 要关闭的块，以及一个新的 list 块 作为孩子打开的 block_quote。 一个 list_item 也是 添加为 的子项 list和 paragraph 作为孩子 这 list_item。 然后，将文本添加到新的 paragraph： 

-> document
  -> block_quote
       paragraph

    -> list (type=bullet tight=true bullet_char=-)
      -> list_item
        -> paragraph
             "Qui *quodsi iracundia*"

-> document

一旦所有输入都被解析，所有打开的块都会被关闭。

然后，我们“走树”，访问每个节点，并解析原始 段落和标题的字符串内容作为内联。 在此 我们已经看到了所有的链接引用定义，所以我们可以 随时解决参考链接。

一种用于解析嵌套强调和链接的算法

到目前为止，内联解析最棘手的部分是处理强调， 强烈的强调、链接和图像。 这是使用以下方法完成的 算法。

当我们解析内联时，我们点击任一

  • 一连串 * 或 _ 字符，或
  • 一个 [ 或 ![

我们插入一个以这些符号作为其文字内容的文本节点，我们 将指向此文本节点的指针添加到 分隔符堆栈 中。

分隔符堆栈 是一个双向链表。 每 元素包含指向文本节点的指针，以及有关

  • 分隔符的类型 （ [， ![， *， _）
  • 分隔符的数量，
  • 分隔符是否为“活动”（所有分隔符都处于活动状态以启动），以及
  • 分隔符是潜在的开盘器，还是潜在的收盘器， 或两者兼而有之（这取决于前面的字符类型 并遵循分隔符）。

当我们击中 ] 字符，我们称之为 寻找链接或图像 程序（见下文）。

当我们到达输入的末尾时，我们称之为 过程强调 程序（见下文），使用 stack_bottom =空。

查找链接或图像

从分隔符堆栈的顶部开始，我们向后看 通过堆栈寻找开口 [ 或 ![ 定界符。

  • 如果我们没有找到一个，我们将返回一个文本节点 ]。

  • 如果我们确实找到了一个，但它没有 处于活动 状态，我们会删除不活跃的 分隔符，并返回文本节点 ]。

  • 如果我们找到一个并且它处于活动状态，那么我们向前解析以查看是否 我们有一个内联链接/图像、参考链接/图像、紧凑引用 链接/图像，或快捷方式引用链接/图像。

    • 如果我们不这样做，那么我们从 分隔符堆叠并返回文本节点 ]。

    • 如果我们这样做，那么

      • 我们返回一个链接或图像节点，其子节点是内联 在开始分隔符指向的文本节点之后。

      • 我们对 进行流程强调 这些内联 [ 器 如 stack_bottom。

      • 我们删除了开始分隔符。

      • 如果我们有一个链接（而不是图像），我们也会设置所有 [ 开始分隔符之前的分隔符设置为 非活动 。 （这 将阻止我们获取链接中的链接。

过程重点

参数 stack_bottom 设定了我们走多远的下限 在 分隔符堆栈 中下降。 如果它为 NULL，我们可以 一直走到底部。 否则，我们之前停止 访问 stack_bottom。

让 current_position 指向 分隔符堆栈 上的元素 就在上面 stack_bottom （或第一个元素，如果 stack_bottom 为 NULL）。

我们跟踪 openers_bottom 对于每个分隔符 键入 （ *， _） 和结束分隔符运行的每个长度 （模 3）。 将其初始化为 stack_bottom。

  • 现在，回头看看堆栈（停留在上面 stack_bottom 和 这 openers_bottom 对于此分隔符类型） 第一个匹配的潜在开瓶器（“匹配”表示相同的分隔符）。

  • 如果找到：

    • 弄清楚我们是强调还是强烈强调： 如果更近的跨度和开端跨度的长度 >= 2，则我们有 强，否则有规律。

    • 删除 1（对于常规 emph）或 2（对于强 emph）分隔符 从开始和结束文本节点。 如果它们变为空 因此，删除它们并删除相应的元素 分隔符堆栈。 如果删除了关闭节点，请重置 current_position 添加到堆栈中的下一个元素。
